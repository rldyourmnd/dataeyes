## 1. Резюме

**DataEyes** — это единый OpenAI-совместимый API-шлюз к десяткам frontier-моделей ведущих вендоров (OpenAI, Google, Anthropic, DeepSeek, Alibaba, Moonshot, ByteDance, MiniMax, Z.AI, Xiaomi) через **один ключ и один интерфейс**, с мульти-региональными точками входа (Global / International / Китай).

Главное коммерческое преимущество — **цена ниже прямого доступа к вендору на все модели**: DataEyes отдаёт каждую модель со скидкой к официальному vendor-прайсу (публично — **от 10% до 90%** в зависимости от модели и тарифа ресурса), плюс действуют **volume-скидки**. Поверх доступа мы даём **объективную систему выбора модели**: открытый бенчмарк-сервис, который измеряет качество, скорость, стоимость и надёжность каждой модели на воспроизводимых задачах и трассирует всё в Langfuse. Это снимает главный риск интеграции LLM — «какую модель брать под задачу и бюджет» — превращая его из догадок в измеримое решение.

**Что получает партнёр:**
- **скидка на все модели** к vendor-прайсу (10–90%) + volume-скидки — прямая экономия бюджета;
- один контракт и один API вместо десятка интеграций с разными вендорами;
- независимые, воспроизводимые данные о качестве/латентности/стоимости моделей;
- готовый слой наблюдаемости и контроля затрат;
- быстрый онбординг (drop-in замена OpenAI-клиента).

## 2. О платформе DataEyes

DataEyes (бренд **数眼 / DataEyes**) — унифицированный интерфейс к множеству LLM. Ключевые свойства (подтверждены технической интеграцией):

- **Единый OpenAI-совместимый API** — `chat/completions`, Responses API, streaming, function calling. Дополнительно поддерживаются **Anthropic-совместимый** и **Gemini-совместимый** форматы — миграция с существующего стека минимальна.
- **Динамический каталог моделей** — `GET /v1/models`: список доступных моделей определяется вашим ключом, без хардкода. Каталог покрывает текст/LLM, мультимодальные, image и video-модели.
- **Web Search / Reader API** — отдельные собственные сервисы живого веб-поиска и парсинга страниц (`api.dataeyes.ai`) для агентных сценариев.
- **Политика приватности** — промпты не логируются и не используются для обучения («your prompts never logged or trained»).
- **Мульти-регион** — несколько точек входа для отказоустойчивости и близости к рынку:

| Регион / назначение | Точка входа |
|---|---|
| Global (Кремниевая долина, прямое подключение) | `https://cloud.dataeyes.ai/v1` |
| International (high-defense, CF-кластер) | `https://platform.dataeyes.ai/v1` |
| Китай (КНР, выделенный CDN) | `https://cloud-cn.shuyanai.com/v1` |
| Web Search / Reader API | `https://api.dataeyes.ai` |

- **Аутентификация** — `Authorization: Bearer <key>` (ключ выдаётся в консоли после регистрации).
- **Документация** — `dataeyes.ai`, `doc.dataeyes.ai`.

> Юридические реквизиты вендора, официальные SLA и сертификации (data residency, ISO/SOC2 и т.п.) уточняются у DataEyes — см. раздел 12.

## 3. Проблема, которую мы закрываем

«Много моделей через один ключ» — это удобство, но в продакшене партнёр сталкивается с типовыми проблемами интеграции. Ниже — реальные классы инцидентов, которые наш бенчмарк выявляет **до** вывода в прод:

| # | Проблема интеграции | Последствие | Как закрываем |
|---|---|---|---|
| 1 | Несовместимость endpoint (например, отсутствие `/v1` в base URL) | Клиент падает на старте | Валидация конфигурации, корректные дефолты |
| 2 | Часть «рекламируемых» моделей недоступна конкретному ключу | Сюрприз в проде | Динамический discovery `/v1/models` |
| 3 | Разное качество tool-calling / function calling между моделями | Невалидные аргументы инструментов | Скоринг tool-accuracy + JSON-fallback |
| 4 | Большой разброс latency (особенно first-token) между моделями | Непредсказуемый UX | Замер p50/p95 латентности |
| 5 | Поле стоимости отсутствует в ответе API | Невозможно атрибутировать затраты | Локальный расчёт по референс-ценам, `cost_status` без фабрикации |
| 6 | Трассировка теряется, если запрос обошёл шлюз | Нет наблюдаемости | Принудительная маршрутизация + Langfuse |
| 7 | Rate-limit (429) при конкурентных вызовах | Сбои нагрузки | Контролируемый параллелизм, лимиты на прогон |
| 8 | «Грязный» JSON (markdown-обёртки, `<think>`-блоки) | Срыв strict-схем | Санитайзинг и извлечение JSON |

**Вывод:** ценность не только в широте каталога, но и в **объективной приёмке моделей** перед интеграцией.

## 4. Что мы предлагаем

1. **Доступ к платформе DataEyes** — единый API к каталогу frontier-моделей (раздел 5), мульти-регион, drop-in совместимость.
2. **Независимый бенчмарк-сервис** (`model-bench`, открытый код) — воспроизводимая оценка моделей по 10 наборам тестов (suites): от smoke-проверок и строгих JSON-схем до многошаговых reasoning-, code- и system-design-задач и 5–10-шагового агентного сценария.
3. **Слой наблюдаемости и экономики** — трассировка каждого вызова в Langfuse (латентность, токены, токены/сек, оценочная стоимость, оценки качества), хранение артефактов и сводных отчётов.
4. **Отчётность под партнёра** — выгрузка результатов прогонов в человекочитаемые отчёты (Markdown/PDF) для принятия решений и презентаций конечным клиентам.

## 6. Доказательство: результаты независимого бенчмарка

Ниже — **иллюстративный прогон** нашего инструмента по реальному каталогу DataEyes: набор `deep_eval` (4 задачи × 100 баллов = 400), по одной флагманской модели на вендора, 9 моделей. Задачи: многошаговый reasoning, генерация кода, строгое следование инструкции, QA по длинному контексту. Структура прогона воспроизводима на вашем ключе — вы получите сопоставимые артефакты по своим моделям и задачам.

| # | Модель | Вендор | Балл / 400 | Ср. латентность/задача |
|---|---|---|---:|---:|
| 1 | gpt-5.4 | OpenAI | 400 | ~9.2 с |
| 1 | gemini-3.1-pro-preview | Google | 400 | ~9.5 с |
| 1 | minimax-m2.5 | MiniMax | 400 | **~5.7 с (самый быстрый)** |
| 1 | glm-5 | Z.AI | 400 | ~23.6 с |
| 1 | doubao-seed-2-0-pro | ByteDance | 400 | ~48.4 с |
| 1 | qwen3.6-plus | Alibaba | 400 | ~56.9 с |
| 7 | claude-opus-4-6 | Anthropic | 390 | ~11.3 с |
| 7 | kimi-k2.5 | Moonshot | 390 | ~58.0 с |
| 9 | deepseek-v3.2 | DeepSeek | 340 | ~22.7 с |

**Ключевые выводы (воспроизводимы):**
- **Качество подтянулось у всех топ-вендоров**: 6 из 9 моделей — идеальные 400/400 на этом наборе.
- **Лучший баланс «скорость × качество»**: `gpt-5.4`, `gemini-3.1-pro-preview`, `minimax-m2.5` — высший балл при низкой латентности (квадрант «быстро и точно»). `gemini-3.1-pro-preview` — лидер по пропускной способности (~200+ выходных токенов/сек).
- **Скрытые риски latency**: `kimi-k2.5` и `qwen3.6-plus` дают высокий балл, но на задаче генерации кода доходят до **1.5–2.5 минут** на ответ — непригодны для интерактивных сценариев без оговорок.
- **Самая сложная задача — QA по длинному контексту**: именно на ней «проседают» более слабые на этом наборе модели (deepseek-v3.2).

> Это точечный замер на фиксированном наборе задач, а не непрерывный SLA-мониторинг. Бенчмарк воспроизводим: любой партнёр может перепрогнать его на своём ключе и своих задачах и получить сопоставимые артефакты.

## 7. Технологические преимущества

- **Цена ниже вендора на все модели.** Скидка к официальному vendor-прайсу на каждую модель (10–90% по каталогу) + volume-скидки — прямая экономия без потери качества.
- **Drop-in совместимость.** OpenAI / Anthropic / Gemini-форматы + Responses API + streaming + function calling — миграция существующего кода минимальна.
- **Приватность данных.** Промпты не логируются и не используются для обучения.
- **Мульти-регион и отказоустойчивость.** Global (Кремниевая долина) / International (CF high-defense) / CN точки входа; прямой fallback на DataEyes при недоступности шлюза.
- **Надёжность прогона.** Изоляция ошибок по моделям (падение одной модели не валит весь прогон — статус `partial`), жёсткий wall-clock timeout на вызов, санитайзинг «грязного» JSON, поддержка thinking-моделей (извлечение reasoning-контента).
- **Наблюдаемость.** Двухуровневая трассировка в Langfuse: токен-уровень (через шлюз LiteLLM) + прикладной уровень (run/case-спаны и оценки). Метаданные: run_id, suite, case_id, модель, вендор.
- **Экономика без фабрикации.** Стоимость считается по прозрачной референс-таблице; если данных нет — честный статус `unknown`, а не выдуманное число.
- **Хранилище и экспорт.** PostgreSQL (прогоны, результаты, события, артефакты) + S3-совместимое хранилище (сырые ответы, отчёты, выгрузки) с локальным fallback.
- **Инженерная зрелость.** Открытый код, тесты (32 проходят), линт (ruff), CI/CD (тесты на Python 3.11/3.12, CodeQL, gitleaks, Dependabot) — прозрачно и проверяемо.

## 9. Как начать (внедрение)

1. **Доступ.** Партнёр получает ключ DataEyes; проверяем доступный каталог через `/v1/models`.
2. **Smoke-проверка.** Запускаем `smoke`-набор — подтверждаем связность, JSON-валидность, базовую латентность.
3. **Профильные наборы.** Подбираем suites под задачи партнёра (`json_schema`, `agent_mini`, `deep_eval`, `fast` и т.д.).
4. **Наблюдаемость.** Подключаем Langfuse — все вызовы, метрики и оценки видны в реальном времени.
5. **Решение.** По сводному отчёту (качество × латентность × стоимость) фиксируем модели под каждый сценарий.

Технически сервис поднимается за день: `make up && make migrate && make dev` (PostgreSQL + хранилище + шлюз), затем `curl /benchmarks/run`. Ориентировочный срок пилота — **`[1–2 недели — согласовать]`**.

## 12. Приложения и оговорки

- **Методология цен (раздел 5).** «Vendor-прайс» — публичная ориентировочная цена вендора (USD/1M токенов, июнь 2026), заложенная в бенчмарк как референс расчёта стоимости (DataEyes в ответах API стоимость не передаёт — она оценивается локально или помечается как недоступная). «Ваша цена» = vendor-прайс × коэффициент ресурса DataEyes (стабильный official-tier) × партнёрское представительство; цифры в разделе 5 рассчитываются из единого источника в коде (`benchsvc.scoring`, скрипт `scripts/compute_partner_pricing.py`), поэтому КП и инструмент согласованы. Диапазоны «до −X%» отражают публичные бейджи скидок dataeyes.ai (например DeepSeek 40–90%, Claude/Gemini 20–70%) и более глубокие special/self-deployed/volume-тарифы. **Точный per-token прайс по вашему ключу и объёму подтверждается в консоли DataEyes / по запросу.**
- **Источник результатов бенчмарка.** Раздел 6 — иллюстративный прогон `deep_eval` по реальному каталогу DataEyes (9 моделей, по одной на вендора). Это точечный замер на фиксированном наборе задач, воспроизводимый на стороне партнёра; он не является непрерывным мониторингом или гарантией SLA.
- **Не подтверждены в материалах и требуют уточнения у DataEyes:** юридические реквизиты вендора, официальные SLA/uptime, сертификации и data residency (GDPR/SOC2/ISO 27001), точные лимиты тарифов, размеры контекстных окон.
- **Инструмент.** Бенчмарк-сервис (`model-bench`) — открытый код в монорепо `dataeyes`; методология, наборы тестов, скоринг и расчёт цен прозрачны и проверяемы.
